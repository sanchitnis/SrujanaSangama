# SrujanaSangama: Design Reference

> Technical and architectural design of the SrujanaSangama platform. This document covers **how** the platform is built — domains, agents, skills, commands, memory architecture, workspace setup, and validators. For **what** the platform does, see `architecture.md`.

**Platform Version:** 0.6 | **Academic Year:** 2026–27
*Author: Dr. Sanjay Chitnis (in collaboration with Claude.ai)*

---

## Overview

SrujanaSangama is a **shared folder of markdown-based agent instructions, skills, and slash commands** for REVA University — deliberately simple rather than a packaged plugin marketplace. There is no installation step and no manifest format to learn. The same folder of plain markdown files is read directly by **Claude Code**, **GitHub Copilot in VS Code**, and **Google Antigravity**, because all three can read instruction files, custom commands, and project context from a folder without any platform-specific packaging.

The one place a genuine cross-platform technical standard is used is the **Model Context Protocol (MCP)** — for the rare cases where a workflow needs a live tool call (for example, pulling current SLCM data). MCP is supported natively by Claude Code, Copilot, and Antigravity, so it is the only piece of "real" integration in the system. Everything else — agents, skills, and workflows — is plain markdown, openly readable, and requires no build step.

The system has three physical roots in every user's workspace:

| Root | Location | Access | Purpose |
|---|---|---|---|
| `SrujanaSangama/` | OneDrive shared folder (REVA IT) | Read-only | Domain folders, skills, commands, validators — the shared intelligence |
| `srujana-memory/` | User's local machine or personal OneDrive | Read-write | Private and collaborative memory — soul, wiki, tasks, projects, portfolios |
| `reva-information/` | OneDrive shared folder (REVA IT) | Read-only | University reference files — regulations, syllabi, policies, accreditation docs |

These three roots are bound together in a single `.code-workspace` file generated by the onboarding domain's first command.

## Domain Architecture

Nine domains plus the cross-domain Personal Productivity layer. Each domain is a folder; there is no plugin handle to install — an agent is simply pointed at the relevant `domains/<name>/` folder, or refers to it by name in natural language (e.g. "use the teaching-learning commands").

| Domain Folder | Primary Users | T.R.A.C.K. Mapping |
|---|---|---|
| `onboarding` | Every new user — run first | Setup |
| `personal-productivity` | Every individual — cross-domain productivity layer | K — Kaizen & Institutional Excellence (cross-domain) |
| `teaching-learning` | Faculty | T — Teaching & Learning |
| `srujana-shodha` | Faculty researchers, PhD supervisors, research scholars | R — Research |
| `admissions-branding` | Admissions office, marketing team | A — Academic Admin (Admissions) |
| `placement-tpc` | TPC coordinators, placement faculty | A — Academic Admin (Placement) |
| `academic-admin` | Faculty, admin staff, IQAC, exam cell, alumni office | A — Academic Admin (all other verticals) |
| `innovator` | Faculty consultants, NEST mentors, IP inventors | C — Consulting & Product Development |
| `kaizen-excellence` | All roles — team through strategic | K — Kaizen & Institutional Excellence (collective + strategic) |
| `strategic-planning` | Deans, VC office, IQAC, REVA Summit team | K — Strategic level |

## Memory Architecture

### `srujana-memory/` — Structure and Purpose

The memory system is the intelligence that persists across all sessions. It lives **outside** the shared SrujanaSangama folder — in the user's personal space — and is read-write throughout usage. It is never uploaded to the shared repository.

```plaintext
srujana-memory/
│
├── srujana.code-workspace               ← Generated by onboarding; binds all three roots
│
├── my-memory/                           ← Private individual memory
│   ├── soul.md                          ← Identity anchor: who I am, my roles, my Ikigai
│   ├── inbox/                           ← Raw, unsorted capture — the front door of the wiki
│   │   └── YYYY-MM-DD-*.md              ← Dumped notes, meeting scraps, article highlights, voice-to-text drops
│   ├── wiki/                            ← Personal LLM-wiki — processed, structured, wiki-linked knowledge
│   │   ├── people/                      ← One entity page per person (colleague, scholar, collaborator)
│   │   ├── concepts/                    ← One entity page per idea, technique, or topic
│   │   ├── courses/                     ← One entity page per course taught
│   │   ├── projects/                    ← One entity page per active or past project
│   │   ├── sources/                     ← Summarised source material (papers, reports, articles read)
│   │   └── _notes/                      ← AI sidecar notes: inferred links, open questions, "investigate later"
│   ├── semantic/                        ← Stable facts about me
│   │   ├── profile.md                   ← Role, school, expertise, qualifications
│   │   ├── teaching.md                  ← Courses, COs, pedagogical preferences
│   │   ├── research.md                  ← Research areas, active grants, publication targets
│   │   └── professional.md              ← Committees, leadership roles, consulting engagements
│   ├── episodic/                        ← Time-stamped events and decisions
│   │   ├── weekly-reviews/              ← Weekly review logs (YYYY-MM-DD.md)
│   │   ├── meeting-notes/               ← BOS, committee, industry meeting notes
│   │   └── decisions/                   ← Significant decisions with rationale
│   ├── context/                         ← What is live and active right now
│   │   ├── this-semester.md             ← Current courses, active projects, deadlines
│   │   ├── tasks.md                     ← GTD task list — capture, clarify, organise, next actions
│   │   └── active-projects.md           ← Running summaries of each active project
│   ├── gps/                             ← Goal-Plan-Sankalpa structure
│   │   ├── sankalpa-current.md          ← Current resolutions/committed intents for the year
│   │   ├── goals-current.md             ← Goals derived from each Sankalpa, with linked Plans

│
```

### Memory Types — Distinction and Use

| Type | Folder | What it contains | How often updated |
|---|---|---|---|
| **Inbox** | `my-memory/inbox/` | Raw, unprocessed capture — dumped without needing to decide structure first | Anytime, continuously |
| **Wiki** | `my-memory/wiki/` | Processed, wiki-linked entity pages compiled from the inbox — compounds over time | After each inbox processing pass |
| **Semantic** | `my-memory/semantic/` | Stable facts: role, expertise, teaching areas, research domains. Changes rarely. | Semester start, role change |
| **Episodic** | `my-memory/episodic/` | Time-stamped events: meeting notes, decisions, weekly reviews. Append-only log. | Weekly or after significant events |
| **Context** | `my-memory/context/` | Live state: current semester load, GTD task list, running projects. Changes frequently. | Weekly or per session |
| **GPS** | `my-memory/gps/` | Sankalpa (resolution), Goals, and the habit log that connects daily action to both. | Semester boundaries; habit log daily/weekly |
| **Performance** | `my-memory/performance/` | Monthly reflection + shareable reports, FDP log, appraisal evidence. | Monthly; appraisal evidence as events occur |
| **Habits** | `my-memory/habits/` | GPS goals, habit tracking, FDP log, Kaizen commitments. | Daily/weekly |
| **Public** | `public-memory/` | Portfolio artefacts for sharing with industry, accreditation bodies, and placement. | After each publication, patent, or project |
| **Collaborative** | `collaborations/` | Shared context for teams, research groups, PhD scholars. Managed externally (OneDrive/Drive/Git). | As the collaboration evolves |

### `soul.md` — The Identity Anchor

`soul.md` is the single most important memory file. Every domain's agent reads it at the start of every session to orient itself. It is compact by design — 200–400 words — and contains:

```markdown
# soul.md — [Name]

## Who I am
Name, designation, school, department, years at REVA.
My core professional identity in 2–3 sentences.

## My Roles (this academic year)
- Primary: [e.g. Associate Professor, BTech CS]
- Additional: [e.g. HOD, BOS Member, PhD Supervisor, IQAC Coordinator]

## What I teach (this semester)
- [Course name] — [semester] — [CO focus]

## Research identity
- Domain: [e.g. Computer Vision, NLP]
- Active grants: [brief]
- Target publications: [brief]

## My Sankalpa this year
The resolution(s) I am seriously committed to — see gps/sankalpa-current.md for full detail.

## My current Goals
Goals derived from my Sankalpa, spanning teaching, research, admin/service, and personal growth — see gps/goals-current.md for full detail with linked Plans.

## Ikigai alignment
What I love, what I am good at, what the world needs, what REVA needs from me.

## Working preferences
How I like to work with AI: tone, depth, format preferences.
```

## Domain Specifications

Each domain follows the same simple structure — markdown only, no manifest:

```plaintext
domains/domain-name/
├── README.md            ← Domain entry point — what it does, read first by the agent
├── rules/                ← Prompt boundary rules (always loaded)
│   └── *.md
├── commands/             ← Slash command implementations, one .md per command
│   └── *.md
└── memory-templates/     ← Templates (only in onboarding and personal-productivity)
    └── *.md
```

If a domain genuinely needs a live tool call (rare — mainly SLCM or examination data lookups), it references the shared `mcp/reva-data-server/` rather than bundling its own server.

### `onboarding`

**Purpose:** The first thing every user runs. Creates `srujana-memory/` with the correct role-specific structure, populates `soul.md` and `public-memory/profile.md` from interview answers, generates the `.code-workspace` file, and orients the user to the domains most relevant to them.

**Rules loaded:** `REVA_ONBOARDING_STANDARD.md`, `MEMORY_CREATION_RULES.md`

**What it does — step by step:**

```
Step 1 — Welcome and orientation
  Explains what SrujanaSangama is and what is about to happen.
  Reads README.md from the shared folder for context.

Step 2 — Role interview (structured conversation)
  Asks: Name, designation, school, department
  Asks: Primary role + any additional roles (multi-select)
  Asks: Courses teaching this semester (name, semester, brief CO focus)
  Asks: Research areas and active projects/grants
  Asks: PhD scholars being supervised (names and topics)
  Asks: Consulting engagements or funded projects
  Asks: Sankalpa (resolutions) and Goals for this academic year (teaching, research, admin, personal)
  Asks: Ikigai — what energises them professionally
  Asks: Working preferences (depth, tone, format)

Step 3 — Public profile interview
  Asks: Qualifications and experience
  Asks: Key publications (top 5)
  Asks: Patents filed or granted
  Asks: Significant projects and outcomes
  Asks: Awards, recognitions, institutional contributions

Step 4 — Folder creation
  Creates srujana-memory/ at Desktop or specified path
  Creates my-memory/ with full subfolder structure, including inbox/ and wiki/
  Creates public-memory/ with subfolder structure
  Creates role-specific collaboration folders (union of all selected roles)
  Writes soul.md from interview answers
  Writes semantic/profile.md, semantic/teaching.md, semantic/research.md
  Writes public-memory/profile.md (CV format)
  Writes context/this-semester.md with current courses and active projects
  Writes gps/sankalpa-current.md and gps/goals-current.md from interview answers
  Writes one starter wiki/ entity page per course, project, and key collaborator named in the interview

Step 5 — Workspace file generation
  Creates srujana.code-workspace in srujana-memory/ with:
    { "folders": [
        { "path": "../SrujanaSangama" },
        { "path": "." },
        { "path": "../reva-information" }
      ]
    }
  Paths are relative to srujana-memory/ location.

Step 6 — Domain orientation
  Based on roles selected, tells the user which domains are most relevant to them.
  Provides the three most useful first commands for their role.
  Instructs them to open srujana.code-workspace to continue.
```

**Slash commands:** `/onboard`, `/update-profile`, `/add-role`, `/add-collaboration`

### `personal-productivity`

**Purpose:** The cross-domain layer every user lives in day to day — a personal knowledge wiki built on the LLM-wiki pattern, a full GTD task engine, GPS goal steering, and performance reflection. Conceptually it belongs to Kaizen & Institutional Excellence, since disciplined self-management is itself a continuous improvement practice, but it reaches across every other domain. It reads `soul.md`, `context/tasks.md`, `gps/`, and `context/this-semester.md` at the start of every session to orient itself without re-briefing.

**Rules loaded:** `REVA_ACADEMIC_STANDARDS.md`, `PERSONAL_REFLECTION_RULES.md`, `MEMORY_UPDATE_PROTOCOL.md`, `LLM_WIKI_PROCESSING_RULES.md`, `GTD_DISCIPLINE_RULES.md`

**Memory templates provided:** Full `my-memory/` template set in `memory-templates/` — used by `onboarding` and by `scripts/local/init.py`.

#### Personal Knowledge Wiki (LLM-wiki pattern, Obsidian-readable)

| Capability | How it works |
|---|---|
| **Inbox capture** | Anything can be dropped into `inbox/` at any time — a pasted meeting note, a half-formed idea, a quote from a paper, a voice memo transcript — with zero filing decision required at capture time |
| **Inbox processing** | `/wiki-process` reads everything new in `inbox/`, extracts concepts, people, courses, and projects mentioned, and either creates new entity pages in `wiki/` or updates existing ones, adding `[[wiki-links]]` between related pages |
| **Entity pages** | Each page in `wiki/people/`, `wiki/concepts/`, `wiki/courses/`, `wiki/projects/`, and `wiki/sources/` is a structured, Wikipedia-style page for one thing, cross-linked to everything related to it |
| **Sidecar notes** | `wiki/_notes/` holds the AI's inferred connections, contradictions noticed, and "this might matter later" observations — a second, looser layer of memory alongside the structured wiki |
| **Obsidian compatibility** | The entire `wiki/` folder is plain markdown with `[[double-bracket links]]`, so it opens directly as an Obsidian vault with no conversion step |
| **Compounding query** | `/wiki-ask <question>` reasons over the full wiki rather than searching raw documents from scratch each time, so answers improve and deepen the more the wiki is used |

#### GTD Task Engine

| Capability | How it works |
|---|---|
| **Capture** | Any new commitment, request, or idea is captured to `context/tasks.md` immediately, regardless of domain |
| **Clarify** | `/task-clarify` turns vague captures into concrete next actions, projects, or "someday/maybe" items |
| **Organise** | Tasks are tagged by domain (T.R.A.C.K.), context, and energy required; reviewed and re-sorted in `/weekly-review` |
| **Reflect** | The weekly and monthly review cycles are where the GTD system is kept trustworthy — nothing survives unexamined for more than a week |
| **Engage** | `/next-actions` surfaces what to actually do right now, filtered by available time and context |

#### GPS Goal Steering (Goal–Plan–Sankalpa)

| Capability | How it works |
|---|---|
| **Sankalpa setting** | `/gps` opens by asking what resolutions — serious, committed intents — the person is making this year, written to `gps/sankalpa-current.md` |
| **Goal derivation** | Each Sankalpa is broken into one or more Goals with a concrete Plan and milestones, written to `gps/goals-current.md` |
| **Traceability check** | At any point, `/gps-trace <task>` shows the chain from that task up through its Plan, its Goal, to the Sankalpa it serves — or flags that no such chain exists |
| **Recalibration** | At semester boundaries, `/gps` reviews progress, energy, and institutional context, and proposes adjustments to Goals or Plans without abandoning the underlying Sankalpa lightly |

#### Performance Management & Monthly Reporting

| Capability | How it works |
|---|---|
| **Monthly report generation** | `/monthly-report` produces one file in `performance/monthly/` with two clearly separated sections: a candid reflection against GPS commitments, and a clean, shareable accomplishment summary |
| **Drift detection** | Compares the month's actual task completion and wiki activity against stated Goals; flags where effort has quietly diverged from intent |
| **Course correction** | Each monthly report closes with an explicit decision per active Goal: continue, adjust, or retire |
| **Appraisal evidence compilation** | `/appraisal-prep` draws on `public-memory/`, `performance/monthly/`, and `performance/fdp-log.md` to assemble a self-appraisal narrative and evidence package for REVA's formal annual appraisal cycle |

#### Cross-Domain Capabilities

| Capability | How it works |
|---|---|
| **Session start** | Reads soul.md + context/ + gps/ → orients itself → confirms current focus with user |
| **Memory update** | At session end, `/session-close` updates tasks.md and active-projects.md |
| **Weekly review** | `/weekly-review` pulls from episodic/, context/, and wiki/ → structured cross-domain review → writes to episodic/weekly-reviews/ |
| **Writing partner** | Drafts across all domains using semantic/ and wiki/ for context and public-memory/ for portfolio evidence |

## Workspace Setup — Step by Step

This is the complete setup sequence for a new user. It is also the content basis for the How to Use guide on aihub.reva.edu.in.

### Step 1 — Link the shared SrujanaSangama folder

REVA IT will share the `SrujanaSangama` OneDrive folder with your REVA email. In your OneDrive:
- Right-click the shared folder notification → **Add shortcut to My Files**
- This makes `SrujanaSangama/` appear in your local OneDrive sync as a read-only folder

### Step 2 — Link the shared reva-information folder

REVA IT will also share the `reva-information` OneDrive folder. Add it to your OneDrive the same way.

### Step 3 — Open the shared folder in your AI tool

**Claude Code:**
```bash
cd ~/OneDrive/SrujanaSangama
claude .
```

**VS Code + GitHub Copilot:**
Open VS Code → File → Open Folder → select `SrujanaSangama`

**Google Antigravity:**
Open Antigravity → Add Folder → select `SrujanaSangama`

### Step 4 — Read README.md and run onboarding

Open `README.md` in the shared folder. Then in the chat panel, simply say:
```
Use the onboarding domain to set me up. Run /onboard.
```
Answer the structured interview. The onboarding agent will:
- Create `srujana-memory/` on your Desktop (or a path you specify)
- Create all memory files and role-specific folders, including your inbox and wiki
- Generate `srujana.code-workspace`

### Step 5 — Open the workspace file

Close the current folder. Open the generated workspace file:

**VS Code:** File → Open Workspace from File → `srujana-memory/srujana.code-workspace`

**Claude Code:**
```bash
claude srujana-memory/srujana.code-workspace
```

You now have all three roots active: `SrujanaSangama` (shared skills and commands), `srujana-memory` (your private memory), `reva-information` (university reference).

### Step 6 — Start working

There is no handle to remember — just tell the agent which domain folder you want to use, in plain language. Your most useful first command, by role:

| Role | Start here |
|---|---|
| **Faculty** | "Use teaching-learning, run /design-course" |
| **Researcher** | "Use srujana-shodha, run /onboarding" |
| **HOD** | "Use academic-admin, run /attainment" then "use kaizen-excellence, run /dept-kaizen" |
| **Dean** | "Use strategic-planning, run /summit-prep" then "use kaizen-excellence, run /school-kaizen" |
| **IQAC** | "Use academic-admin, run /iqac-dashboard" then "run /aqar-generate" |
| **TPC** | "Use placement-tpc, run /readiness-audit" |
| **Admissions** | "Use admissions-branding, run /enrolment-analytics" |
| **All** | "Use personal-productivity, run /weekly-review" every Monday morning |

## Web Guide — "How to Use" (aihub.reva.edu.in)

The aihub.reva.edu.in/srujana-sangama page is an instructions-only onboarding guide — no code, no interactivity beyond standard web navigation. Its job is purely to get a faculty member from "I've heard about SrujanaSangama" to "my workspace is open and I just ran my first command," using the exact sequence in **Workspace Setup — Step by Step** above, restated in plain language with role-based starting points.

**What it contains:**
- A 30-second explanation of what SrujanaSangama is and is not (not a chatbot product, not an installable app — a shared folder of instructions used through tools the person may already have)
- The six setup steps, written for someone who has never linked a shared OneDrive folder or opened a multi-root workspace before
- A role picker ("I am a: Faculty / Researcher / HOD / Dean / IQAC Coordinator / TPC / Admissions staff") that jumps to the relevant "start here" row from the Step 6 table
- Screenshots or short screen-recordings of each setup step on Windows OneDrive, VS Code, and Claude Code
- A troubleshooting section for the most likely failure points: shared folder not syncing, workspace file not opening, onboarding not finding the shared folder path

**What it deliberately does not contain (for now):** any interactive prompt generator, memory file upload, or AI call of its own. That richer experience — a browsable capability explorer with prompt generation for people without any of the three agent harnesses installed — remains a valid future idea, but is explicitly out of scope until this simpler guide is live and in use.
